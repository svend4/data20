## Философия системы

### Принципы
1. **Атомарность:** Каждый документ должен освещать одну конкретную тему
2. **Связность:** Документы связаны через перекрестные ссылки и метаданные
3. **Поисковость:** Легко найти нужную информацию через индексы, теги и ключевые слова
4. **Масштабируемость:** Система работает как с 10, так и с 10,000 документами
5. **Автоматизация:** Максимум процессов автоматизировано для работы с AI/скриптами

### Аналогия: Дерево знаний
Представьте базу знаний как дерево:
- **Корень** - главный индекс (INDEX.md)
- **Ствол** - основные категории (computers, household, cooking)
- **Ветви** - подкатегории (hardware, software, programming)
- **Листья** - конкретные статьи
- **Плоды** - практическая польза от информации

Каждый новый факт должен найти свое место на правильной ветке.

## Жизненный цикл информации

### Этап 1: Поступление (Inbox)
**Цель:** Собрать сырую информацию из различных источников

**Процесс:**
1. Новая информация попадает в `inbox/raw/`
2. Создается файл с префиксом даты: `YYYY-MM-DD-название.md`
3. Используется шаблон входящего материала
4. Статус: `processing_status: new`

**Пример файла:**
```
inbox/raw/2026-01-02-новости-процессоров.md
```

### Этап 2: Первичная обработка
**Цель:** Определить, куда относится информация

**Задачи:**
1. **Прочитать** материал
2. **Определить** основные темы
3. **Разбить** на логические части, если материал большой
4. **Определить** целевые разделы в базе знаний
5. **Переместить** в `inbox/processing/`
6. **Статус:** `processing_status: in_progress`

**Чек-лист первичной обработки:**
- [ ] Определена основная тема
- [ ] Определены подтемы
- [ ] Материал разбит на части (если нужно)
- [ ] Найдены целевые разделы в knowledge/
- [ ] Проверены существующие статьи на похожие темы
- [ ] Определены ключевые слова (tags)

### Этап 3: Структурирование
**Цель:** Интегрировать информацию в базу знаний

**Решения:**
1. **Дополнить существующую статью** - если тема уже есть
2. **Создать новую статью** - если тема новая
3. **Разбить на несколько статей** - если материал большой и охватывает разные темы

**Правило разбиения:**
- Если статья >500 строк → рассмотреть разбиение
- Если статья охватывает >3 подтем → создать отдельные статьи
- Создать индексную статью, связывающую части

**Процесс интеграции:**
1. Найти целевую папку (`knowledge/{category}/articles/{subcategory}/`)
2. Создать или обновить статью
3. Заполнить метаданные (frontmatter)
4. Добавить перекрестные ссылки на связанные статьи
5. Обновить индексный файл категории
6. Переместить исходник в `inbox/processed/`

### Этап 4: Связывание
**Цель:** Создать сеть связанных знаний

**Типы связей:**
1. **Прямые ссылки** - в тексте статьи: `[[related-article]]`
2. **Метаданные** - в frontmatter: `related: [article1.md, article2.md]`
3. **Теги** - общие ключевые слова
4. **См. также** - секция в конце статьи

**Пример:**
```markdown
## См. также
- [[python-patterns]] - Паттерны программирования
- [[clean-code]] - Чистый код
- [[testing]] - Тестирование
```

### Этап 5: Индексирование
**Цель:** Обновить индексы для быстрого поиска

**Что обновляется:**
1. **Индекс категории** (`knowledge/{category}/index/INDEX.md`)
2. **Главный индекс** (`INDEX.md`)
3. **Статистика** (количество статей, дата обновления)

**Автоматизация:**
```bash
python tools/update_indexes.py
```

### Этап 6: Ревизия и архивирование
**Цель:** Поддерживать актуальность базы знаний

**Процесс:**
- **Раз в 3 месяца:** Проверка устаревших статей
- **Раз в год:** Полная ревизия категории

**Критерии архивирования:**
- Информация устарела (технология не используется)
- Появилась более актуальная статья
- Статья не обновлялась >2 лет и потеряла актуальность

**Процесс архивирования:**
1. Добавить пометку `status: archived` в frontmatter
2. Переместить в `archive/YYYY-MM/`
3. Удалить из индексов
4. Обновить ссылки в других статьях

## Правила организации контента

### Иерархия структуры

#### Уровень 1: Категория (Домен)
**Примеры:** computers, household, cooking
**Критерий:** Широкая область знаний, не пересекающаяся с другими

#### Уровень 2: Подкатегория
**Примеры:** programming, hardware, appliances
**Критерий:** Логическая группировка внутри категории

#### Уровень 3: Тема
**Примеры:** python-patterns, cpu-comparison
**Критерий:** Конкретная статья по определенному вопросу

### Правило "Колибри и цветка"
> Как каждому виду колибри подходит только определенный вид цветка, так и каждая информация должна найти своё идеальное место в иерархии.

**Алгоритм размещения:**
1. Определить широкую категорию (computers/household/cooking)
2. Определить подкатегорию (programming/appliances/recipes)
3. Проверить существующие статьи на похожие темы
4. Если похожая тема есть → дополнить её
5. Если нет → создать новую статью
6. Если статья становится слишком большой → разбить на подтемы

### Принцип DRY (Don't Repeat Yourself)
**Правило:** Не дублировать информацию
**Решение:** Вместо дублирования использовать ссылки

**Плохо:**
```
article1.md - описание Docker
article2.md - описание Docker (повтор)
```

**Хорошо:**
```
article1.md - полное описание Docker
article2.md - ссылка на article1.md + специфичная информация
```

### Правило разбиения больших статей

**Когда разбивать:**
- Статья >1000 строк
- Статья охватывает >5 подтем
- Сложно найти конкретную информацию

**Как разбивать:**
1. Создать папку вместо файла
2. Создать INDEX.md с оглавлением
3. Разбить на тематические статьи
4. Связать через перекрестные ссылки

**Пример:**
```
# Было
programming/python-complete-guide.md (2000 строк)

# Стало
programming/python/
├── INDEX.md                    # Оглавление
├── basics.md                   # Основы
├── patterns.md                 # Паттерны
├── async.md                    # Асинхронность
├── testing.md                  # Тестирование
└── best-practices.md           # Лучшие практики
```

## Система метаданных

### Обязательные поля
```yaml
---
title: "Название статьи"           # Заголовок
date: 2026-01-02                   # Дата создания/обновления
tags: [тег1, тег2, тег3]           # Минимум 3-5 тегов
category: категория                # computers/household/cooking
subcategory: подкатегория          # programming/appliances/etc
status: draft|published|archived   # Статус
---
```

### Дополнительные поля
```yaml
source: "Источник информации"
related: [article1.md, article2.md]
author: "Автор"
date_updated: 2026-01-15
version: 1.2
```

### Правила тегирования

**Типы тегов:**
1. **Категориальные** - широкая область (компьютеры, бытовая-техника)
2. **Тематические** - конкретная тема (Python, холодильник)
3. **Функциональные** - назначение (руководство, обзор, инструкция)
4. **Технологические** - конкретные технологии (Docker, React, NVMe)

**Рекомендации:**
- Минимум 3-5 тегов на статью
- Максимум 10-12 тегов
- Использовать kebab-case: `машинное-обучение`
- Быть конкретным: вместо `программирование` → `python`, `javascript`

**Примеры хороших тегов:**
```yaml
tags: [Python, паттерны-проектирования, ООП, архитектура, best-practices]
```

**Примеры плохих тегов:**
```yaml
tags: [полезное, интересное, важное]  # Слишком общие
```

## Автоматизация обработки

### Для ИИ-ассистентов (Claude, GPT, etc.)

**Ключевая информация для AI:**
1. **Frontmatter метаданные** - структурированные данные для парсинга
2. **Заголовки** - иерархия информации
3. **Теги** - быстрая категоризация
4. **Секция "См. также"** - связанные темы

**Промпт для обработки входящих материалов:**
```
Проанализируй материал в inbox/raw/FILENAME.md:
1. Определи основную тему и подтемы
2. Найди существующие статьи на похожие темы в knowledge/
3. Определи, нужно создать новую статью или дополнить существующую
4. Предложи структуру интеграции
5. Сгенерируй метаданные (теги, category, subcategory)
6. Найди связанные статьи для перекрестных ссылок
```

**Промпт для создания статьи:**
```
Создай статью на основе материала:
1. Используй шаблон из docs/TEMPLATES.md
2. Заполни все метаданные
3. Структурируй содержание логично
4. Добавь примеры (если применимо)
5. Добавь перекрестные ссылки на связанные статьи
6. Создай секцию "См. также"
7. Добавь секцию "Ключевые моменты"
```

### Скрипты автоматизации

**Инструменты (в папке tools/):**

1. **process_inbox.py** - Обработка входящих материалов
   - Сканирует inbox/raw/
   - Анализирует содержимое
   - Предлагает категории и теги
   - Создает заготовки статей

2. **update_indexes.py** - Обновление индексов
   - Сканирует все статьи
   - Обновляет индексы категорий
   - Обновляет главный индекс
   - Считает статистику

3. **find_related.py** - Поиск связанных статей
   - Анализирует теги
   - Ищет похожие темы
   - Предлагает перекрестные ссылки

4. **validate.py** - Валидация
   - Проверяет наличие обязательных метаданных
   - Проверяет корректность ссылок
   - Проверяет структуру папок

**Использование:**
```bash
# Обработать входящие
python tools/process_inbox.py

# Обновить все индексы
python tools/update_indexes.py

# Найти связанные статьи для article.md
python tools/find_related.py knowledge/computers/articles/ai/llm.md

# Валидировать базу знаний
python tools/validate.py
```

## Best Practices

### Для написания статей

1. **Структура:**
   - Краткое описание в начале (2-3 предложения)
   - Логичная иерархия заголовков
   - Ключевые моменты в конце
   - Ссылки и источники

2. **Язык:**
   - Простой и понятный
   - Конкретные примеры
   - Избегать жаргона (или объяснять его)
   - Писать для целевой аудитории

3. **Форматирование:**
   - Используйте списки для перечислений
   - Используйте таблицы для сравнений
   - Используйте код-блоки для примеров кода
   - Выделяйте важные моменты **жирным**

### Для организации

1. **Регулярная ревизия:**
   - Раз в месяц: проверить inbox/
   - Раз в квартал: обновить статистику
   - Раз в год: полная ревизия категории

2. **Контроль качества:**
   - Проверять метаданные при добавлении
   - Валидировать ссылки
   - Обновлять устаревшую информацию

3. **Резервное копирование:**
   - Git для версионирования
   - Регулярные коммиты
   - Описательные сообщения коммитов

### Для работы с тегами

1. **Консистентность:**
   - Использовать единый формат (kebab-case)
   - Проверять существующие теги перед созданием новых
   - Не создавать синонимы (выбрать один вариант)

2. **Иерархия:**
   - От общего к частному
   - Пример: `[компьютеры, программирование, Python, Django]`

3. **Практичность:**
   - Теги должны помогать в поиске
   - Избегать слишком общих тегов (`полезное`, `интересное`)
   - Избегать слишком узких тегов (использованных только 1 раз)

## Работа с большими объемами информации

### Стратегия масштабирования

**Когда категория разрастается (>100 статей):**
1. Создать дополнительные подкатегории
2. Разбить INDEX.md на несколько индексов
3. Использовать вложенные папки

**Пример:**
```
# Было
knowledge/computers/articles/programming/
├── python1.md
├── python2.md
├── python3.md
... (50 файлов)

# Стало
knowledge/computers/articles/programming/
├── python/
│   ├── INDEX.md
│   ├── basics/
│   ├── advanced/
│   └── frameworks/
├── javascript/
└── go/
```

### Поиск информации

**Методы поиска:**
1. **Через индексы** - начать с INDEX.md
2. **Через теги** - использовать grep/поиск по тегам
3. **Через полнотекстовый поиск** - grep по содержимому
4. **Через связанные статьи** - переходить по ссылкам

**Команды для поиска:**
```bash
# Найти все статьи с тегом "Python"
grep -r "tags:.*Python" knowledge/

# Найти все статьи о Docker
grep -r "Docker" knowledge/ --include="*.md"

# Найти все индексы
find knowledge/ -name "INDEX.md"
```

## Контроль версий (Git)

### Правила коммитов

**Формат сообщения:**
```
[Категория] Краткое описание

Подробное описание (опционально)
```

**Примеры:**
```bash
git commit -m "[computers] Добавлена статья про LLM модели 2026"
git commit -m "[household] Обновлен гид по выбору холодильников"
git commit -m "[docs] Обновлена методология работы с базой"
git commit -m "[inbox] Обработаны материалы от 2026-01-02"
```

### Workflow

**Ежедневная работа:**
```bash
# Добавить новую информацию в inbox
git add inbox/
git commit -m "[inbox] Добавлены новости по AI"

# Обработать и интегрировать
python tools/process_inbox.py
git add knowledge/
git commit -m "[computers] Интегрированы новости по AI"

# Обновить индексы
python tools/update_indexes.py
git add INDEX.md knowledge/*/index/
git commit -m "[docs] Обновлены индексы"

# Отправить на GitHub
git push
```

## Метрики качества

### KPI базы знаний

1. **Покрытие тем:**
   - Количество категорий: 3+
   - Количество подкатегорий: 15+
   - Количество статей: 100+

2. **Качество метаданных:**
   - 100% статей имеют frontmatter
   - 90%+ статей имеют 3+ тега
   - 80%+ статей имеют перекрестные ссылки

3. **Актуальность:**
   - 90%+ статей обновлялись за последний год
   - 0% статей со статусом `draft` старше 1 месяца
   - Inbox обрабатывается еженедельно

4. **Связность:**
   - Среднее количество ссылок на статью: 3+
   - Количество "островных" статей (без ссылок): <10%

## Заключение

Эта методология - живой документ. По мере роста базы знаний и появления новых практик, методология должна обновляться.

**Ключевые принципы:**
- Структурируйте информацию иерархически
- Используйте метаданные для поиска
- Связывайте знания через ссылки
- Автоматизируйте рутинные задачи
